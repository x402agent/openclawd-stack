## CRITICAL Findings

### C-1. Insufficient Transaction Signature Verification (x402)

- **File:** `x402/src/facilitator.ts` ~L65-82
- **OWASP:** Cryptographic Failures
- **Issue:** Signature verification only checks that the signature object exists and the pubkey matches. It does **not** cryptographically verify the signature using `nacl.sign.detached.verify()`.
- **Impact:** An attacker could submit a forged transaction with a valid-looking but unverified signature, potentially stealing funds.
- **Fix:** Deserialize and cryptographically verify the Ed25519 signature before accepting payment.

### C-2. No Nonce Replay Protection (x402)

- **File:** `x402/src/server.ts` ~L88-105
- **OWASP:** Software & Data Integrity Failures
- **Issue:** Nonces are generated and sent to clients but never tracked server-side. The same signed transaction can be replayed indefinitely.
- **Impact:** Double-spend / replay of valid payment transactions.
- **Fix:** Maintain a `Map<nonce, timestamp>` with TTL. Reject any nonce that has been seen before.

### C-3. SSRF via Webhook URLs (Telegram Bot)

- **File:** `telegram-bot/src/api/server.ts` ~L596-610
- **OWASP:** Server-Side Request Forgery (SSRF)
- **Issue:** User-supplied webhook URLs are validated for protocol (`http:`/`https:`) but not for hostname. Private IPs (`127.0.0.1`, `169.254.169.254`, `10.0.0.0/8`, `192.168.0.0/16`) are allowed.
- **Impact:** Attacker can target internal services, cloud metadata endpoints (AWS/GCP IMDS), or exhaust resources.
- **Fix:** Blocklist reserved/private IP ranges. Resolve DNS and validate the resolved IP before delivering webhooks.

### C-4. Optional Webhook Secret Silently Disables HMAC (Telegram Bot)

- **File:** `telegram-bot/src/api/webhooks.ts` ~L20
- **OWASP:** Security Misconfiguration
- **Issue:** If `WEBHOOK_SECRET` env var is unset, webhooks are sent without HMAC signatures. Receivers cannot verify authenticity.
- **Impact:** Webhook forgery — attackers on the network can inject fake events.
- **Fix:** Fail startup or loudly warn if `WEBHOOK_SECRET` is empty. Never send unsigned webhooks.

### C-5. Insecure Randomization for Fee Recipients (Core SDK)

- **File:** `src/bondingCurve.ts` ~L254, `src/fees.ts` ~L136
- **OWASP:** Cryptographic Failures
- **Issue:** Uses `Math.random()` (not CSPRNG) to select fee recipients. Output is predictable.
- **Impact:** Fee distribution can be predicted and gamed by front-running transactions.
- **Fix:** Use `crypto.getRandomValues()` or `crypto.randomInt()`.

## HIGH Findings

### H-1. Token Transfer Instruction Not Fully Validated (x402)

- **File:** `x402/src/facilitator.ts` ~L48-60
- **Issue:** Only checks that SPL Token program is invoked with non-empty data. Does not verify: recipient ATA, transfer amount, or token mint (USDC).
- **Impact:** Attacker could submit a transaction that transfers the wrong token or wrong amount.
- **Fix:** Decode the SPL Token transfer instruction and validate destination, mint, and amount.

### H-2. Blockhash Not Revalidated Before Settlement (x402)

- **File:** `x402/src/facilitator.ts` ~L95-136
- **Issue:** No `isBlockhashValid()` check before submitting. Expired blockhash causes silent failure.
- **Impact:** Failed settlements, potential fund lock.
- **Fix:** Call `connection.isBlockhashValid()` before `sendRawTransaction()`.

### H-3. Weak API Key Hashing (Telegram Bot)

- **File:** `telegram-bot/src/api/server.ts` ~L308-316
- **Issue:** Uses a simple 32-bit JSHash for client identification. Collisions are trivial.
- **Impact:** Client impersonation via hash collision.
- **Fix:** Use `crypto.createHash('sha256')`.

### H-4. No Authentication on WebSocket Relay

- **File:** `websocket-server/src/server.ts` ~L77-95
- **Issue:** Any client can connect without authentication. No connection limits per IP.
- **Impact:** DoS via connection flooding.
- **Fix:** Add per-IP connection limits (e.g., max 10) and a global max (e.g., 1000).

### H-5. Open CORS on WebSocket Server

- **File:** `websocket-server/src/server.ts` ~L14-18
- **Issue:** `Access-Control-Allow-Origin: *` allows any origin.
- **Impact:** Any website can connect and consume the relay data.
- **Fix:** Restrict to known origins or add token-based auth.

### H-6. Unsafe Type Assertions (Core SDK)

- **Files:** `src/sdk.ts` ~L104,117,125,171; `src/onlineSdk.ts` ~L78,913,925
- **Issue:** Multiple `null as any` casts bypass TypeScript type safety.
- **Impact:** Potential runtime errors from unexpected null access.
- **Fix:** Use proper typing or guard clauses.

### H-7. Unhandled Promise Rejections (Channel Bot)

- **File:** `channel-bot/src/index.ts` ~L48-62
- **Issue:** Async event handlers lack try/catch. Unhandled rejections terminate the process.
- **Impact:** Bot crashes silently on fetch/post errors.
- **Fix:** Wrap all async callbacks in try/catch.

### H-8. XSS via innerHTML in Live Dashboards

- **Files:** `live/index.html` ~L636,752; `live/trades.html` ~L801
- **Issue:** Direct `innerHTML` assignment with string interpolation from external data (token names, symbols).
- **Impact:** Stored XSS if malicious token metadata is rendered.
- **Fix:** Use `textContent` for user data or sanitize with DOMPurify.

## MEDIUM Findings

### M-1. BN.toNumber() in Analytics (Core SDK)

- **Files:** `src/analytics.ts` ~L146,204,252; `src/tokenIncentives.ts` ~L39,48,54
- **Issue:** `.toNumber()` on BN values. Safe for bounded values (basis points, day indices) but fragile.
- **Fix:** Add comments justifying safety. Consider `BigInt` for future-proofing.

### M-2. PumpFun API Response Not Schema-Validated (WebSocket)

- **File:** `websocket-server/src/solana-monitor.ts` ~L82-104
- **Issue:** `JSON.parse()` result assumed to match expected shape.
- **Fix:** Add Zod schema validation.

### M-3. Solana RPC URL Not Validated (Telegram Bot)

- **File:** `telegram-bot/src/config.ts` ~L20-30
- **Issue:** Accepts any URL. Could point to attacker-controlled endpoint.
- **Fix:** Enforce HTTPS, optionally whitelist known RPC providers.

### M-4. Client IP Spoofable via X-Forwarded-For (WebSocket)

- **File:** `websocket-server/src/server.ts` ~L85
- **Issue:** Trusts `x-forwarded-for` header unconditionally.
- **Fix:** Only trust behind a known proxy; otherwise use `socket.remoteAddress`.

### M-5. Missing CSP Headers (All HTML)

- **Files:** `live/index.html`, `live/trades.html`, `live/vanity.html`
- **Issue:** No Content-Security-Policy headers.
- **Fix:** Add CSP meta tags or headers.

### M-6. Facilitator URL SSRF (x402)

- **File:** `x402/src/server.ts` ~L139
- **Issue:** `facilitatorUrl` not validated for private/reserved IPs.
- **Fix:** Validate URL against reserved IP blocklist.

### M-7. Missing Request Body Size Limits (All Servers)

- **Issue:** No documented body size limits on HTTP endpoints.
- **Fix:** Add `Content-Length` limits (e.g., 1MB max).

### M-8. No Rate Limiting on Health/OpenAPI Endpoints (Telegram Bot)

- **File:** `telegram-bot/src/api/server.ts` ~L195-206
- **Issue:** `/health` and `/openapi` skip auth and rate limiting.
- **Fix:** Apply rate limiting before auth checks.

### M-9. Error Object Logging Without Sanitization (All)

- **Issue:** `log.error('Error:', err)` across components may leak stack traces, IPs, paths.
- **Fix:** Sanitize error objects before logging.

### M-10. Empty Bot Token Allowed at Startup (Telegram Bot)

- **File:** `telegram-bot/src/config.ts` ~L12
- **Issue:** Falls back to empty string if `TELEGRAM_BOT_TOKEN` unset in API-only mode.
- **Fix:** Validate non-empty token whenever bot features are used.

## LOW Findings

### L-1. `catch (err: any)` Pattern (Multiple Files)

- **Issue:** Generic error catching without type discrimination.
- **Fix:** Use `unknown` type, narrow with `instanceof Error`.

### L-2. Non-Null Assertions on Array Access (Core SDK)

- **Files:** `src/bondingCurve.ts` ~L254, `src/fees.ts` ~L136
- **Issue:** `array[index]!` without bounds check. Safe but fragile.

### L-3. Overpayment Accepted Without Warning (x402)

- **File:** `x402/src/server.ts` ~L112-113
- **Issue:** `payment.amount >= required` allows overpayment.
- **Fix:** Add tolerance band or exact match.

### L-4. Console.log Instead of Structured Logger (WebSocket)

- **File:** `websocket-server/src/server.ts`
- **Issue:** All logging via `console.log()`. No levels, timestamps, or parseable format.

### L-5. `cargo audit` Not Available in Dev Container

- **Issue:** Rust dependency auditing tool not installed.
- **Fix:** Add `cargo install cargo-audit` to dev container setup.
